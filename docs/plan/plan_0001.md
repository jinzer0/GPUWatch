# GPUWatcher v0.1 구현 계획서

## 1. 문서 목적과 계획 원칙

이 문서는 `docs/draft/draft_0001.md`의 구현 기획을 실제 개발 가능한 v0.1 실행 계획으로 정리한 문서이다.

GPUWatcher v0.1의 핵심 목표는 macOS 데스크톱 앱에서 여러 Linux NVIDIA GPU 서버의 현재 GPU 상태를 확인하는 것이다. 구현 계획은 가장 빠르고 안전하게 "원격 서버 1대의 실제 GPU snapshot을 앱에 표시하는 경로"를 먼저 증명한 뒤, 저장, polling, 에러 처리, 전체 화면 완성으로 확장한다.

계획 원칙은 다음과 같다.

- 첫 번째 실제 live snapshot을 가장 이른 핵심 마일스톤으로 둔다.
- 프로토콜, 저장소, SSH 실행, scheduler 불변식은 UI polish보다 먼저 고정한다.
- v0.1은 내부 MVP로 제한하고, MVP 외 기능을 구현하지 않는다.
- Python collector, Rust/Tauri backend, SQLite, React UI 사이의 경계를 명확히 둔다.
- React는 raw collector JSON을 직접 파싱하지 않는다.
- Rust/Tauri backend가 protocol validation, parsing, persistence, read model 생성을 책임진다.
- 실패한 poll은 health/error metadata만 갱신하고, 마지막 성공 snapshot을 덮어쓰지 않는다.
- 알 수 없거나 수집 불가능한 metric은 `0`으로 표시하지 않고 `null` 또는 unknown으로 다룬다.
- 각 단계는 demo gate와 verification gate를 모두 가져야 한다.

## 2. v0.1 Locked Decisions

| 항목 | v0.1 결정 |
|---|---|
| 지원 플랫폼 | macOS Desktop only |
| Desktop framework | Tauri |
| Frontend | React, TypeScript, Tailwind CSS |
| Frontend server/backend state | TanStack Query |
| Frontend local UI state | Zustand only for ephemeral UI state |
| Backend | Rust Tauri commands and backend services |
| Local storage | SQLite |
| Remote access | system `ssh` |
| Remote auth | SSH key-based auth only |
| Credential storage | SSH key path만 저장. password, passphrase, private key material 저장 금지 |
| Collector | Python CLI package `gpuwatcher` |
| Collector command | `gpuwatcher --json` |
| Data transport | JSON over SSH stdout |
| Protocol | `protocolVersion`/`schemaVersion`이 있는 JSON protocol v0.1 |
| Snapshot retention | latest successful validated snapshot only |
| GPU busy rule | `processCount > 0` 또는 `memoryUsedMiB > 512`이면 busy, 아니면 free |
| Unknown metrics | `0`으로 치환하지 않고 `null`/unknown으로 표시 |
| Polling | per-server single-flight, global concurrency cap |
| First proof | 원격 서버 1대의 live snapshot을 앱 backend 경로로 수집, 저장, 표시 |

v0.1에서는 remote collector가 원격 서버에 이미 설치되어 있고 `PATH`에서 `gpuwatcher`로 실행 가능하다고 가정한다. 필요하면 서버 설정에 collector command path를 명시할 수 있게 계획한다.

## 4. 산출물과 컴포넌트 경계

v0.1 산출물은 두 개이다.

| 산출물 | 역할 |

| Python remote collector `gpuwatcher` | 원격 Linux NVIDIA 서버에서 현재 GPU/process 상태를 수집하고 JSON으로 stdout 출력 |

컴포넌트 경계는 다음과 같다.

| 컴포넌트 | 책임 | 하지 말아야 할 일 |
|---|---|---|
| React UI | 화면 렌더링, 사용자 입력, Tauri command 호출 | raw collector JSON parsing, SSH 실행, SQLite 직접 접근 |
| TanStack Query | Tauri command 결과 cache, server/backend state 동기화 | 장기 persistence 대체 |
| Zustand | 선택된 탭, modal 상태, sort UI 등 ephemeral UI state | 서버 상태 source of truth 역할 |
| Tauri Commands | frontend/backend typed boundary | 복잡한 비즈니스 로직 직접 보유 |
| Rust backend services | server CRUD, protocol parser, command runner, scheduler, SQLite repository, read model 생성 | UI rendering 책임 |
| SQLite | server config, health/status, latest success snapshot 저장 | GPU/process history 저장 |
| Command Runner | local/fake command와 system SSH 실행 abstraction | shell string interpolation |
| System SSH Runner | 원격 `gpuwatcher --json` 실행 | password/passphrase handling UI |
| Poll Scheduler | per-server polling lifecycle 관리 | 한 서버 실패를 전체 앱 실패로 전파 |
| Python Collector | NVML/psutil 기반 GPU/process snapshot 생성 | 앱 상태 저장, SSH 처리, UI 결정 |
| Protocol fixtures | collector/backend/frontend 계약 검증 | 임의 mock data drift 허용 |

Tauri command는 thin, typed, async orchestration layer로 유지한다. 긴 작업은 Rust async service에서 처리하고, 공유 service/repository/scheduler는 Tauri managed state로 관리한다.

## 5. 데이터/프로토콜 계약

### 5.1 Protocol 원칙

Collector와 desktop app 사이의 계약은 JSON protocol v0.1이다. `gpuwatcher --json` 실행 시 stdout에는 정확히 하나의 JSON object만 출력한다. 로그, 경고, diagnostic message는 stderr로만 출력한다.

모든 정상/collector-level error 응답은 다음 필드를 포함한다.

| 필드 | 설명 |
|---|---|
| `protocolVersion` | collector/app protocol version. v0.1에서는 `1` |
| `schemaVersion` | JSON schema version. v0.1에서는 `1` |
| `ok` | collector 실행 결과. `true` 또는 `false` |
| `timestamp` | remote host에서 collector가 관측한 timestamp |

Desktop app은 unsupported `protocolVersion` 또는 `schemaVersion`을 명확한 protocol/collector error로 표시한다. v0.1은 multi-version negotiation을 구현하지 않는다.

### 5.2 Success envelope

`ok: true` 응답은 다음 데이터를 포함한다.

| 필드 | 설명 |
|---|---|
| `server` | remote hostname, driver version, CUDA version 등 payload metadata |
| `gpus` | GPU 목록 |
| `warnings` | optional. 부분 수집 실패나 unavailable metric 설명 |

GPU field의 단위는 field name에 포함한다. 예: `memoryUsedMiB`, `temperatureCelsius`, `powerDrawWatt`, `gpuUtilizationPercent`.

Metric이 수집 불가능하면 `null`로 둔다. 알 수 없는 값을 `0`으로 대체하지 않는다.

### 5.3 Collector error envelope

`ok: false` 응답은 collector가 실행되었지만 collector 내부에서 실패한 경우에만 사용한다.

예시는 다음과 같다.

```json
{
  "protocolVersion": 1,
  "schemaVersion": 1,
  "ok": false,
  "timestamp": "2026-06-01T15:24:00+09:00",
  "error": {
    "type": "collector_dependency_unavailable",
    "message": "NVML is not available"
  }
}
```

SSH timeout, auth failure, unreachable host, collector missing, malformed stdout은 collector JSON이 아니다. 이들은 desktop backend가 app-level error로 분류한다.

### 5.4 Error taxonomy

| Layer | Error examples | 생성 위치 |
|---|---|---|
| `transport_ssh` | `ssh_timeout`, `ssh_auth_failed`, `ssh_unreachable`, `ssh_host_key_failed` | Rust SSH runner |
| `remote_command` | `collector_missing`, `remote_command_failed`, `remote_command_timeout` | Rust command runner |
| `protocol` | `protocol_malformed_json`, `protocol_schema_invalid`, `protocol_unsupported_version` | Rust protocol parser |
| `collector` | `collector_dependency_unavailable`, `nvml_unavailable`, `collector_internal_error` | Python collector JSON |
| `storage_app` | `sqlite_error`, `migration_error`, `snapshot_write_failed` | Rust repository |
| `unknown` | 분류 불가능한 오류 | Rust backend |

UI는 error layer를 구분해서 표시한다. transport/command/protocol error는 latest success snapshot을 지우지 않는다.

### 5.5 Timestamp semantics

| Timestamp | 생성 위치 | 용도 |
|---|---|---|
| `timestamp` | Python collector | remote payload metadata 표시용 |
| `poll_started_at` | desktop app | poll attempt 시작 시간 |
| `poll_finished_at` | desktop app | poll attempt 종료 시간 |
| `received_at` | desktop app | valid snapshot 수신 시간 |
| `last_success_at` | desktop app | freshness/stale 판단 기준 |

Freshness, stale, offline 판단은 app-local timestamp를 기준으로 한다. Remote host clock skew 때문에 collector timestamp를 stale 판단 기준으로 사용하지 않는다.

### 5.6 Free/busy GPU rule

v0.1 MVP rule은 다음과 같다.

- GPU가 하나 이상의 process를 가지면 busy이다.
- Process가 없더라도 `memoryUsedMiB > 512`이면 busy이다.
- 위 조건에 해당하지 않으면 free이다.
- `memoryUsedMiB`가 unavailable이면 process count 기준만 사용하고, unavailable 값을 `0`으로 취급하지 않는다.
- Threshold `512 MiB`는 v0.1에서 고정하며, v0.2 이후 설정 가능 후보로 둔다.

### 5.7 Fixture catalog

Protocol fixture는 collector, Rust parser, frontend read model 테스트의 공통 입력이다.

| Fixture | 목적 |
|---|---|
| `success_single_gpu.json` | 단일 GPU 정상 응답 |
| `success_multi_gpu.json` | 다중 GPU 정상 응답 |
| `success_no_gpu.json` | GPU 없는 서버 |
| `success_no_processes.json` | GPU는 있으나 process 없음 |
| `success_partial_metrics.json` | fan/power/process metric 일부 `null` |
| `collector_nvml_unavailable.json` | NVML unavailable collector error |
| `collector_error.json` | 일반 collector error JSON |
| `malformed_json.txt` | stdout이 JSON이 아님 |
| `missing_required_field.json` | required field 누락 |
| `unsupported_protocol_version.json` | unsupported protocol/schema version |
| `ssh_timeout.case` | SSH timeout 분류용 |
| `ssh_auth_failed.case` | SSH auth failure 분류용 |
| `ssh_unreachable.case` | host unreachable 분류용 |
| `collector_missing.case` | `gpuwatcher` PATH/command missing 분류용 |

## 6. SSH/Auth/Command Execution 계약

v0.1은 system `ssh`를 사용한다. embedded SSH library는 v0.1에서 도입하지 않는다.

### 6.1 Auth assumptions

- 사용자는 terminal에서 key-based SSH 접속이 가능해야 한다.
- 앱은 password authentication을 지원하지 않는다.
- 앱은 passphrase를 저장하지 않는다.
- passphrase-protected key는 OS ssh-agent 또는 기존 terminal SSH 환경에서 처리되어야 한다.
- host key trust는 사용자의 기존 SSH/known_hosts 정책을 따른다.
- host key failure는 connection failure로 표시하고, v0.1에서 host-key management UI를 제공하지 않는다.

### 6.2 Remote command policy

v0.1 기본 command는 다음이다.

```bash
gpuwatcher --json
```

서버 설정에는 collector command path를 override할 수 있는 field를 둘 수 있다. 예: `/home/user/.local/bin/gpuwatcher --json`. 단, command 실행은 shell string 조합이 아니라 안전한 argument-vector 구성으로 처리해야 한다.

### 6.3 Command safety

반드시 지켜야 할 규칙은 다음과 같다.

- `ssh` 실행은 argument-vector process spawning으로 수행한다.
- host, port, username, key path, remote command를 shell string으로 이어붙이지 않는다.
- `BatchMode=yes` 또는 이에 준하는 noninteractive 실행을 사용한다.
- connect timeout과 command timeout을 구분한다.
- timeout 시 child process를 종료한다.
- stdout/stderr capture size limit을 둔다.
- stderr는 error detail로 저장하되 UI에는 안전하게 요약한다.
- command result는 typed `PollResult`로 변환한 뒤 storage/UI로 전달한다.

### 6.4 SSH error mapping

| 상황 | App error |
|---|---|
| 접속 timeout | `ssh_timeout` |
| 인증 실패 | `ssh_auth_failed` |
| host unreachable/DNS failure | `ssh_unreachable` |
| host key 문제 | `ssh_host_key_failed` |
| remote command not found | `collector_missing` |
| non-zero exit | `remote_command_failed` |
| stdout malformed JSON | `protocol_malformed_json` |
| schema validation 실패 | `protocol_schema_invalid` |
| collector `ok: false` | collector error type 그대로 mapping |

## 7. 로컬 저장소(SQLite) 범위와 스키마

SQLite는 v0.1의 유일한 durable local storage이다. Tauri SQLite 사용 시 explicit migration과 capability/permission 설정을 함께 계획한다.

### 7.1 Persistence 원칙

- 서버 설정은 재시작 후 유지되어야 한다.
- latest successful snapshot은 재시작 후 유지되어야 한다.
- failed poll은 latest successful snapshot을 삭제하거나 덮어쓰지 않는다.
- GPU/process history table은 만들지 않는다.
- process table은 latest snapshots에서 backend가 생성하는 read model이다.

### 7.2 Logical schema

`servers`

| Field | 설명 |
|---|---|
| `id` | app 내부 server id |
| `name` | 사용자 표시 이름 |
| `host` | SSH host |
| `port` | SSH port |
| `username` | SSH username |
| `ssh_key_path` | local SSH key path |
| `collector_command` | 기본 `gpuwatcher --json` 또는 configured path |
| `polling_interval_seconds` | per-server interval |
| `enabled` | polling 대상 여부 |
| `config_revision` | edit/delete/disable 시 stale poll discard용 revision |
| `created_at` | app-local timestamp |
| `updated_at` | app-local timestamp |

`server_health`

| Field | 설명 |
|---|---|
| `server_id` | server id |
| `status` | `disabled`, `idle`, `polling`, `online`, `stale`, `offline`, `error` |
| `last_error_type` | latest attempt error type |
| `last_error_message` | latest attempt error message |
| `last_poll_started_at` | app-local poll start |
| `last_poll_finished_at` | app-local poll finish |
| `last_success_at` | latest successful snapshot time |

`latest_snapshots`

| Field | 설명 |
|---|---|
| `server_id` | server id |
| `protocol_version` | validated protocol version |
| `schema_version` | validated schema version |
| `received_at` | app-local received timestamp |
| `raw_json` | latest successful collector JSON |
| `parsed_summary_json` | Overview/Detail/Process 생성을 위한 summary cache |

`schema_migrations`

| Field | 설명 |
|---|---|
| `version` | migration version |
| `applied_at` | migration applied timestamp |

### 7.3 Read model contract

Rust backend는 raw snapshot을 파싱해 다음 DTO를 제공한다.

| DTO | 사용 화면 |
|---|---|
| `ServerOverviewDto` | Overview |
| `ServerDetailDto` | Server Detail |
| `GpuCardDto` | Server Detail |
| `ProcessRowDto` | Process Table |
| `ServerHealthDto` | Overview, Settings |
| `ConnectionTestResultDto` | Settings |

React는 DTO만 렌더링한다. React는 `raw_json`을 읽거나 parsing하지 않는다.

## 8. Polling/Scheduler 규칙

Scheduler는 서버별 독립 polling을 관리한다.

### 8.1 Core invariants

- 서버 하나에는 동시에 하나의 poll만 실행된다.
- poll이 진행 중인 서버에 다음 tick이 오면 skip하거나 reschedule한다.
- 전체 동시 poll 수에는 global concurrency limit을 둔다.
- disabled 서버는 poll하지 않는다.
- server edit/delete/disable 시 진행 중 poll 결과는 cancel하거나 invalid 처리한다.
- app shutdown 시 진행 중 작업을 정리한다.
- 한 서버의 timeout/offline은 다른 서버 poll을 막지 않는다.
- failed poll은 health/error metadata만 갱신한다.
- latest successful snapshot은 stale 상태로 계속 표시할 수 있다.

### 8.2 Revision handling

각 poll은 시작 시점의 `server_id`와 `config_revision`을 캡처한다. 완료 시점에 다음 조건 중 하나라도 참이면 결과를 저장하지 않는다.

- server가 삭제됨
- server가 disabled 됨
- `config_revision`이 변경됨
- app shutdown 과정에서 invalidate 됨

이를 통해 stale completion이 최신 설정을 덮어쓰지 못하게 한다.

### 8.3 Status semantics

| Status | 의미 |
|---|---|
| `disabled` | 사용자가 polling 비활성화 |
| `idle` | enabled지만 현재 poll 중 아님 |
| `polling` | 현재 poll 중 |
| `online` | latest poll 성공, current snapshot 표시 가능 |
| `stale` | 이전 성공 snapshot은 있으나 최근 poll 실패 또는 오래됨 |
| `offline` | SSH/transport 실패, 마지막 성공 snapshot은 있을 수 있음 |
| `error` | protocol/collector/storage 등 명확한 오류 |

## 9. UI 화면별 MVP 수용 기준

### 9.1 Overview

Overview는 등록된 모든 서버의 요약 상태를 보여준다.

수용 기준:

- 서버 이름을 표시한다.
- online/offline/stale/error/disabled 상태를 표시한다.
- GPU 총 개수를 표시한다.
- busy GPU 개수를 표시한다.
- free GPU 개수를 표시한다.
- 평균 GPU utilization을 표시한다.
- 평균 memory usage를 표시한다.
- 최고 온도를 표시한다.
- 마지막 성공 업데이트 시간을 표시한다.
- 최신 poll 실패 시에도 마지막 성공 snapshot을 stale로 표시한다.
- 한 서버의 실패가 다른 서버 표시를 막지 않는다.

### 9.2 Server Detail

Server Detail은 특정 서버의 GPU 상세 정보를 보여준다.

수용 기준:

- server name, host, collector hostname을 구분해서 표시한다.
- driver version, CUDA version을 표시한다.
- last success time과 latest error를 구분해서 표시한다.
- GPU card list를 표시한다.
- GPU index, name, UUID를 표시한다.
- utilization, memory total/used/free, temperature, power draw, fan speed를 표시한다.
- unavailable metric은 `unknown` 또는 `-`로 표시한다.
- GPU별 process list를 표시한다.

### 9.3 Process Table

Process 화면은 모든 서버의 latest snapshots에서 process row를 flatten해서 보여준다.

수용 기준:

- server name을 표시한다.
- GPU index를 표시한다.
- username을 표시한다.
- PID를 표시한다.
- command를 표시한다.
- GPU memory usage를 표시한다.
- GPU utilization, CPU usage, host memory usage를 표시한다.
- unavailable metric은 `0`이 아니라 unknown으로 표시한다.
- 기본 정렬은 GPU memory usage descending이다.
- stale server의 process row는 stale 표시를 가진다.

### 9.4 Settings

Settings는 서버 등록 및 설정 관리를 담당한다.

수용 기준:

## 12. 리스크와 대응

| Risk | Impact | Response |
|---|---|---|
| Protocol drift between Python/Rust/TS | parser/UI 불일치 | schema와 shared fixtures를 Phase 1에서 고정 |
| SSH PATH에서 `gpuwatcher`를 못 찾음 | live snapshot 실패 | server setup doc, collector command path 설정, `collector_missing` error |
| SSH prompt로 scheduler hang | 앱 polling 정지 | noninteractive SSH, timeout, kill-on-timeout, BatchMode-style option |
| passphrase key 처리 혼란 | auth failure | ssh-agent/existing terminal auth를 v0.1 prerequisite로 문서화 |
| NVML/psutil metric 일부 unavailable | 잘못된 0 표시 | null/unknown 표시, partial warnings, optional fields |
| failed poll이 last success를 덮어씀 | stale data 손실 | latest-success storage invariant와 transaction test |
| edit/delete 후 stale poll result 저장 | 잘못된 server 상태 | config_revision stale completion discard |
| offline server가 전체 앱을 느리게 함 | UX 저하 | per-server single-flight, global concurrency limit, timeout |
| UI가 raw JSON parsing | logic drift | backend-owned DTO contract 강제 |
| SQLite capability/migration 누락 | 앱 실행 오류 | Tauri capability/permission setup과 migration test |
| MVP scope creep | 일정 지연 | Non-goals/anti-bloat guardrails 유지 |
